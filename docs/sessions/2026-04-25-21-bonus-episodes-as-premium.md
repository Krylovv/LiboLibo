# 2026-04-25-21 — Bonus-эпизоды помечаются как премиум

## Контекст

Пользователь заметил: в подкасте «Два по цене одного» последний эпизод —
бонусный (для подписчиков), но `audio_url` отдаётся всем, mp3 проигрывается
без подписки.

## Диагноз

Гейтинг по `audio_url` в `api/src/lib/serialize.ts:71-72` работает правильно:
если `isPremium=true` и у viewer'а нет entitlement'а — `audio_url` зануляется.
Проблема — в том, что бонусный эпизод записан в БД с `isPremium=false`.

Единственный признак премиальности в коде сейчас — эпизод присутствует в
Transistor API, но отсутствует в публичном RSS (`api/src/transistor/refresh.ts`,
`syncPremiumFromBulk`). Transistor положил bonus-эпизод **в публичный RSS**
(стандартное поведение для bonus-типа), а наш парсер не читал
`<itunes:episodeType>` — поэтому эпизод сохранился как обычный.

## Что сделали

1. `api/src/transistor/parser.ts` — парсер теперь читает
   `<itunes:episodeType>` и нормализует к `"full" | "trailer" | "bonus" | null`.
2. `api/src/transistor/refresh.ts` — при upsert public-эпизодов помечаем
   `isPremium = (episodeType === "bonus")`. У Либо-Либо bonus-эпизоды — всегда
   контент для подписчиков.
3. `api/test/parser.test.ts` — добавлен кейс для bonus и проверка
   `episodeType` для обычных эпизодов.

Тесты (`vitest`) и `tsc` зелёные.

## Открытые вопросы

- Существующие записи в БД с `isPremium=false` для уже синканных bonus-эпизодов
  поправятся при следующем **fresh** RSS-fetch (если RSS вернёт 304 — upsert
  пропустится). Чтобы исправить разово, можно:
  - сбросить `etag/lastModified` в `feed_fetch` для нужных подкастов и дождаться
    cron-tick, или
  - запустить `npm run refresh` против прод-БД, или
  - подождать любого обновления RSS у соответствующих подкастов.
